## 1. Overview and Motivation

This document describes the upgraded SHACL shapes file (`shapes_upgraded.ttl`) produced for the AudiovisualOntology knowledge graph. The original file was critiqued for validating only **range constraints** ŌĆö checking what a property points *to* ŌĆö while completely ignoring domain constraints, class disjointness, string quality, and referential reachability.

The upgraded file addresses all four gaps. It is organized into three sections:

- **Section A ŌĆö Class Disjointness:** prevents any individual from belonging to two domain classes simultaneously.
- **Section B ŌĆö Domain Validation:** ensures every property is used only by its declared domain class, using `sh:targetSubjectsOf` shapes.
- **Section C ŌĆö Range, Cardinality, and Format:** the original constraints enhanced with `sh:minLength`, `sh:datatype xsd:integer` for year properties, and inverse path reachability checks.

> **Key result:** Running the upgraded file against `out.ttl` exposed multiple real data integrity violations across two validation rounds. All violations were traced to concrete mapping gaps and fixed at the R2RML level. The final validation passes with `sh:conforms true`.

## 2. What the Original File Was Missing

### 2.1 Range-only validation

The original file contained only `NodeShapes` with `sh:targetClass`. These shapes check that when a property appears on an individual of the target class, its value is of the correct type. For example:

```turtle
sh:property [
    sh:path ont:isClassifiedAs ;
    sh:class ont:ContentType ;
] .
```

This confirms that if a `Title` has `ont:isClassifiedAs`, it points to a `ContentType`. It says nothing about whether `ont:isClassifiedAs` could be misused on a `Genre` or a `Role` node.

### 2.2 The domain gap

Without domain shapes, the following violations are undetectable:

- A `Genre` individual with `ont:primaryTitle` asserted on it.
- A `Role` individual with `ont:isClassifiedAs` asserted on it.
- A `Participation` individual with `ont:hasGenre` asserted on it.
- Any mapping error that emits a property on the wrong class of subject.

In a well-formed knowledge graph generated by a correct R2RML mapping, these errors may not occur in practice. But SHACL's purpose is to make the rules explicit and machine-verifiable ŌĆö not to assume the mapping is correct.

### 2.3 Other missing constraints

| Missing Constraint | Risk Without It |
|---|---|
| `sh:minLength 1` on name properties | An empty string `""` passes as a valid `genreName` or `primaryTitle` |
| `sh:datatype xsd:integer` on year properties | A value like `"movie"` or `"N/A"` passes as a valid year |
| `sh:not` class disjointness | The same IRI typed as both `ont:Title` and `ont:Talent` passes silently |
| Inverse path reachability | Orphaned `Participation` or `AlternativeTitle` individuals are invisible |

## 3. Section A ŌĆö Class Disjointness

Three `NodeShapes` enforce that the principal classes are mutually exclusive. Each shape targets one class and uses `sh:not` blocks to forbid any other domain class from being simultaneously asserted on the same individual.

**Shapes defined:** `TitleDisjointShape`, `TalentDisjointShape`, `ParticipationDisjointShape`.

The remaining seven classes (Genre, Category, Role, ContentType, AlternativeTitle, Region, TitleType) are dictionary/leaf classes with no outgoing object properties, making cross-typing extremely unlikely. The three shapes above cover the most critical entities.

### 3.1 Pattern

```turtle
shape:TitleDisjointShape a sh:NodeShape ;
    sh:targetClass ont:Title ;
    sh:not [ sh:class ont:Talent ] ;
    sh:not [ sh:class ont:Participation ] ;
    sh:not [ sh:class ont:Genre ] ;
    sh:not [ sh:class ont:ContentType ] ;
    sh:not [ sh:class ont:AlternativeTitle ] ;
    sh:not [ sh:class ont:Category ] ;
    sh:not [ sh:class ont:Role ] ;
    sh:not [ sh:class ont:Region ] ;
    sh:not [ sh:class ont:TitleType ] .
```

### 3.2 What this catches

If a mapping error causes the same IRI to receive two `rdf:type` assertions ŌĆö for example a `Participation` IRI also typed as `ont:Title` ŌĆö `TitleDisjointShape` fires on that individual. Without this shape, such an error produces no SHACL violation and is only detectable by manually inspecting the data.

## 4. Section B ŌĆö Domain Validation

This is the most significant addition. For every ontology property, a dedicated `NodeShape` targets all subjects of that property (`sh:targetSubjectsOf`) and asserts that the subject must be of the declared domain class (`sh:class`). If any node uses the property without belonging to the correct class, the validator reports a violation.

### 4.1 SHACL mechanism: `sh:targetSubjectsOf`

Standard `sh:targetClass` shapes fire on every individual of a class. The `sh:targetSubjectsOf` selector fires on every individual that appears as the **subject** of at least one triple using the specified property. This makes it ideal for domain checking.

```turtle
shape:DomainHasGenreShape a sh:NodeShape ;
    sh:targetSubjectsOf ont:hasGenre ;
    sh:class ont:Title ;
    sh:message "ont:hasGenre used on a non-Title subject. Domain is ont:Title." .
```

This shape fires on every node `?x` such that a triple `?x ont:hasGenre ?y` exists. It then checks that `?x` is of type `ont:Title`. If `?x` is a `Talent` or a `Genre`, the shape fails.

> **This shape caught a real violation in round 2:** titles with NULL `start_year` were excluded from `MapTitle` by the SQL filter, so they were never typed as `ont:Title`. But `MapTitleGenre` still emitted `ont:hasGenre` on their IRIs ŌĆö which `DomainHasGenreShape` correctly flagged as a domain violation. See Section 7.2 for the full explanation.

## 5. Section C ŌĆö Enhanced Range, Cardinality, and Format

The original constraints are retained and enhanced with three additions: `sh:minLength` guards on all name properties, `sh:datatype xsd:integer` on `ont:startYear`, and inverse path reachability checks on `Participation` and `AlternativeTitle`.

### 5.1 `sh:minLength` ŌĆö preventing empty strings

The original file checked `sh:datatype xsd:string`, which accepts any string including the empty string `""`. The upgraded file adds `sh:minLength 1` to every mandatory name property.

```turtle
sh:property [
    sh:path ont:primaryTitle ;
    sh:minCount 1 ;
    sh:maxCount 1 ;
    sh:datatype xsd:string ;
    sh:minLength 1 ;
    sh:message "A Title must have exactly one non-empty primaryTitle."
] ;
```

This applies to: `ont:primaryTitle`, `ont:talentName`, `ont:alternativeTitleName`, `ont:categoryName`, `ont:roleName`, `ont:genreName`, `ont:content_type_name`, `ont:regionName`, `ont:title_type_name`.

### 5.2 `sh:datatype xsd:integer` ŌĆö correct typing for `startYear`

The original file declared `ont:startYear` as `xsd:string`. This was a workaround for potential dirty data (e.g. `\N` MySQL export markers). The professor correctly pointed out that the right solution is to filter at the SQL level rather than weaken the type.

The database was inspected and confirmed to contain only valid 4-digit years and SQL `NULL` values ŌĆö no `\N` strings:

```sql
SELECT start_year, COUNT(*) FROM title GROUP BY start_year ORDER BY start_year;
-- Result: NULL (8 rows), 1912, 1923, 1936, ... (valid integers only)
```

The SHACL constraint was updated to enforce the correct datatype with semantic bounds:

```turtle
sh:property [
    sh:path ont:startYear ;
    sh:datatype xsd:integer ;
    sh:minInclusive 1888 ;
    sh:maxInclusive 2030 ;
    sh:message "startYear must be an integer between 1888 and 2030."
] ;
```

The lower bound `1888` is semantically grounded ŌĆö it is the year of the oldest surviving film (*Roundhay Garden Scene*). The R2RML mapping was updated to filter NULLs at the SQL level and cast the column as integer:

```turtle
# MapTitle logicalTable ŌĆö NULL rows excluded at SQL level
WHERE start_year IS NOT NULL

# Datatype changed from xsd:string to xsd:integer
rr:objectMap [ rr:column "start_year" ; rr:datatype xsd:integer ]
```

The OWL ontology `DataPropertyRange` for `ont:startYear` was also updated from `xsd:string` to `xsd:integer`, keeping the ontology, mapping, and SHACL fully consistent.

### 5.3 Inverse path reachability

Two classes ŌĆö `Participation` and `AlternativeTitle` ŌĆö are bridge/junction entities that must always be reachable from a parent entity. An orphaned individual of either class indicates a mapping error.

The upgraded `ParticipationShape` adds two inverse path checks:

```turtle
# Every Participation must be referenced by at least one Title
sh:property [
    sh:path [ sh:inversePath ont:contains ] ;
    sh:minCount 1 ;
    sh:class ont:Title ;
    sh:message "A Participation must be referenced by at least one Title via ont:contains."
] ;

# Every Participation must be referenced by at least one Talent
sh:property [
    sh:path [ sh:inversePath ont:participatesIn ] ;
    sh:minCount 1 ;
    sh:class ont:Talent ;
    sh:message "A Participation must be referenced by at least one Talent via ont:participatesIn."
] ;
```

`AlternativeTitleShape` adds:

```turtle
# Every AlternativeTitle must be referenced by at least one Title
sh:property [
    sh:path [ sh:inversePath ont:isAka ] ;
    sh:minCount 1 ;
    sh:class ont:Title ;
    sh:message "An AlternativeTitle must be referenced by at least one Title via ont:isAka."
] ;
```

## 7. Violations Found and How They Were Fixed

The validation process ran in two rounds. Each round exposed real data integrity gaps that were invisible to the original SHACL file.

---

### Round 1 ŌĆö Orphaned Participation (missing Talent)

#### 7.1 The violation

```turtle
sh:focusNode <.../participation/tt8998108_nm8037508_8> ;
sh:resultMessage "A Participation must be referenced by
                  at least one Talent via ont:participatesIn." ;
sh:resultPath [ sh:inversePath ont:participatesIn ] ;
```

#### 7.2 Root cause

`MapParticipation` created `participation/tt8998108_nm8037508_8` from a row in `title_principal` where `talent_id = nm8037508`. However, `nm8037508` was absent from the `talent` table, so `MapTalent` never created a `Talent` individual and never emitted the `ont:participatesIn` triple. The result was a dangling `Participation` node with no owning `Talent`.

#### 7.3 Fix ŌĆö INNER JOIN against `talent` in `MapParticipation`

```turtle
# BEFORE
<#MapParticipation> a rr:TriplesMap ;
    rr:logicalTable [ rr:tableName "title_principal" ] ;

# AFTER
<#MapParticipation> a rr:TriplesMap ;
    rr:logicalTable [ rr:sqlQuery """
        SELECT tp.title_id, tp.talent_id, tp.ord, tp.category_id, tp.job
        FROM title_principal tp
        INNER JOIN talent t ON tp.talent_id = t.talent_id
    """ ] ;
```

---

### Round 2 ŌĆö Orphaned Participations, AlternativeTitles, and Domain Violations (NULL `start_year` filter not propagated)

#### 7.4 The violations

After fixing Round 1 and adding the `WHERE start_year IS NOT NULL` filter to `MapTitle`, a second round of violations appeared:

```turtle
# Domain violation ŌĆö ont:hasGenre on an untyped IRI
sh:focusNode <.../title/tt3814770> ;
sh:resultMessage "ont:hasGenre used on a non-Title subject." ;

# Orphaned Participation ŌĆö no Title pointing to it
sh:focusNode <.../participation/tt3814770_nm2051751_6> ;
sh:resultMessage "A Participation must be referenced by at least one Title via ont:contains." ;

# Orphaned AlternativeTitle ŌĆö no Title pointing to it
sh:focusNode <.../alternativeTitle/tt4092002_2> ;
sh:resultMessage "An AlternativeTitle must be referenced by at least one Title via ont:isAka." ;
```

#### 7.5 Root cause

`MapTitle` used `WHERE start_year IS NOT NULL`, which correctly excluded 8 titles with no release year from becoming `ont:Title` individuals. However, the five dependent mappings ŌĆö `MapTitleGenre`, `MapParticipation`, `MapAlternativeTitle`, `MapAkaTitleTypeLink`, and `MapParticipationPlaysRole` ŌĆö still operated on the full tables. They generated individuals for those 8 excluded titles, but since no `ont:Title` individual existed for them, none of the back-links (`ont:contains`, `ont:isAka`) were ever emitted.

This produced three violation types simultaneously:

- `DomainHasGenreShape` fired because `ont:hasGenre` appeared on an IRI with no `rdf:type ont:Title`
- The inverse `ont:contains` check fired on orphaned `Participation` individuals
- The inverse `ont:isAka` check fired on orphaned `AlternativeTitle` individuals

#### 7.6 Fix ŌĆö propagate the NULL filter to all dependent mappings

The `WHERE start_year IS NOT NULL` condition was propagated to all five dependent mappings via INNER JOINs against the `title` table:

```turtle
# MapTitleGenre
rr:logicalTable [ rr:sqlQuery """
    SELECT tg.title_id, tg.genre_id
    FROM title_genre tg
    INNER JOIN title t ON tg.title_id = t.title_id
    WHERE t.start_year IS NOT NULL
""" ] ;

# MapParticipation (also keeps talent INNER JOIN from Round 1)
rr:logicalTable [ rr:sqlQuery """
    SELECT tp.title_id, tp.talent_id, tp.ord, tp.category_id, tp.job
    FROM title_principal tp
    INNER JOIN talent t ON tp.talent_id = t.talent_id
    INNER JOIN title ti ON tp.title_id = ti.title_id
    WHERE ti.start_year IS NOT NULL
""" ] ;

# MapAlternativeTitle
rr:logicalTable [ rr:sqlQuery """
    SELECT ta.title_id, ta.ord, ta.aka_title, ta.region_id
    FROM title_aka ta
    INNER JOIN title t ON ta.title_id = t.title_id
    WHERE t.start_year IS NOT NULL
""" ] ;

# MapAkaTitleTypeLink
rr:logicalTable [ rr:sqlQuery """
    SELECT tatt.title_id, tatt.ord, tatt.title_type_id
    FROM title_aka_title_type tatt
    INNER JOIN title t ON tatt.title_id = t.title_id
    WHERE t.start_year IS NOT NULL
""" ] ;

# MapParticipationPlaysRole
rr:logicalTable [ rr:sqlQuery """
    SELECT tp.title_id, tp.talent_id, tp.ord, tr.role_id
    FROM title_principal tp
    JOIN talent_role tr ON tp.talent_id = tr.talent_id
    INNER JOIN title ti ON tp.title_id = ti.title_id
    WHERE ti.start_year IS NOT NULL
""" ] ;
```

#### 7.7 Design principle confirmed

The `WHERE start_year IS NOT NULL` filter in `MapTitle` acts as a **dataset scope definition** ŌĆö it defines which titles are considered valid individuals in the knowledge graph. Every mapping that depends on a title's existence must enforce the same scope. Failing to propagate it is equivalent to a missing referential integrity constraint in a relational database.

## 9. How to Run the Validation

### 9.1 Apache Jena CLI

```bash
shacl validate --shapes shapes_upgraded.ttl --data out.ttl
```

If `shacl` is not on the system PATH, use the full path:

```bash
C:\path\to\apache-jena-6.0.0\bat\shacl.bat validate --shapes shapes_upgraded.ttl --data out.ttl
```

### 9.2 Save the report to a file

```bash
shacl validate --shapes shapes_upgraded.ttl --data out.ttl > report.ttl
```

### 9.3 Query violations via SPARQL (Fuseki)

Load both `out.ttl` and `shapes_upgraded.ttl` into the Fuseki dataset, then run:

```sparql
PREFIX sh: <http://www.w3.org/ns/shacl#>
SELECT ?focusNode ?path ?message WHERE {
    ?result a sh:ValidationResult ;
            sh:focusNode ?focusNode ;
            sh:resultMessage ?message .
    OPTIONAL { ?result sh:resultPath ?path }
}
```

### 9.4 Expected result

```turtle
[ rdf:type sh:ValidationReport ;
  sh:conforms true ] .
```

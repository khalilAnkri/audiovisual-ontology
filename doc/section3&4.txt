1. R2RML FUNDAMENTALS
1.1 What is R2RML?
Definition:
R2RML (RDB to RDF Mapping Language) is a W3C Recommendation published in September 2012. It defines a language for expressing customized mappings from relational databases to RDF datasets.
Key Characteristics:

Declarative: You describe what to map, not how to do it
Standard: W3C Recommendation (international standard)
Database-agnostic: Works with any SQL database
Triple-based: Maps database rows to RDF triples

Why R2RML?

Automates DB → RDF transformation
Repeatable process (regenerate anytime DB changes)
Preserves relational integrity in semantic form
Industry standard (tools available from multiple vendors)


1.2 Core R2RML Concepts
TriplesMap
The fundamental building block. One TriplesMap = one transformation rule.
Structure:
turtle<#MapName> a rr:TriplesMap;
    rr:logicalTable [ ... ];     # SQL source
    rr:subjectMap [ ... ];        # IRI generation + type
    rr:predicateObjectMap [ ... ] # Properties
Analogy: Think of a TriplesMap like a recipe:

Logical table = ingredients (where data comes from)
Subject map = plate setup (how to identify each dish)
Predicate-object maps = assembly instructions (what goes where)


Logical Table
Two options:
Option 1: Direct table reference
turtlerr:logicalTable [ rr:tableName "genre" ]

Simple
Maps entire table
Most common for dictionary/reference tables

Option 2: SQL Query
turtlerr:logicalTable [ 
    rr:sqlQuery """
        SELECT tp.title_id, tp.talent_id, tp.ord, tr.role_id
        FROM title_principal tp
        JOIN talent_role tr ON tp.talent_id = tr.talent_id
    """ 
]

Powerful
Can join multiple tables
Custom transformations
Used when schema doesn't perfectly match ontology

When to use SQL query:

Need columns from multiple tables
Need to compute derived values
Need to filter rows
Need to handle complex relationships


Subject Map
Defines how to create the IRI (unique identifier) for each RDF resource.
Template syntax:
turtlerr:subjectMap [
    rr:template "https://khalilankri.../data/genre/{genre_id}";
    rr:class ont:Genre
]
Key components:

rr:template: URI pattern with placeholders {column_name}
rr:class: RDF type (equivalent to rdf:type ont:Genre)

Template rules:

Placeholders replaced with column values
Must produce valid absolute IRIs
Case-sensitive column names
Can combine multiple columns: {title_id}_{talent_id}_{ord}

Example transformation:
Database row: genre_id=1, genre_name="Action"
Generated IRI: https://khalilankri.../data/genre/1
Generated triple: <...data/genre/1> a ont:Genre .

Predicate-Object Map
Defines properties (predicates) and their values (objects).
Three types of objects:
1. Column-based (literals):
turtlerr:predicateObjectMap [
    rr:predicate ont:genreName;
    rr:objectMap [ rr:column "genre_name" ]
]
Generates: <...genre/1> ont:genreName "Action" .
2. Template-based (direct IRIs) - AVOID:
turtlerr:objectMap [ 
    rr:template "https://khalilankri.../data/genre/{genre_id}" 
]
⚠️ Problem: Creates IRI but no rdf:type assertion = "homeless individual"
3. Parent Triples Map (recommended for relationships):
turtlerr:objectMap [
    rr:parentTriplesMap <#MapGenre>;
    rr:joinCondition [ 
        rr:child "genre_id"; 
        rr:parent "genre_id" 
    ]
]
✓ Better: Ensures referenced entity exists with proper type

1.3 Parent Triples Maps vs Direct Templates
This is critical - likely to be asked about!
Direct Template Approach (BAD)
turtle<#MapTitle> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "title" ];
    rr:subjectMap [
        rr:template "https://khalilankri.../data/title/{title_id}";
        rr:class ont:Title
    ];
    rr:predicateObjectMap [
        rr:predicate ont:isClassifiedAs;
        rr:objectMap [ 
            rr:template "https://khalilankri.../data/contentType/{content_type_id}" 
        ]
    ] .
Problems:

Homeless individuals: IRI created but no rdf:type ont:ContentType
Dangling references: What if content_type_id=999 doesn't exist in content_type table?
No validation: No guarantee referenced entity exists
Duplicate URI logic: URI template repeated in multiple places

Result in RDF:
turtle<...title/tt001> a ont:Title ;
    ont:isClassifiedAs <...contentType/2> .

# But <...contentType/2> has NO rdf:type!
# SPARQL query: "Show all ContentTypes" returns NOTHING

Parent Triples Map Approach (GOOD)
turtle<#MapContentType> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "content_type" ];
    rr:subjectMap [
        rr:template "https://khalilankri.../data/contentType/{content_type_id}";
        rr:class ont:ContentType
    ];
    rr:predicateObjectMap [
        rr:predicate ont:content_type_name;
        rr:objectMap [ rr:column "content_type_name" ]
    ] .

<#MapTitle> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "title" ];
    rr:subjectMap [
        rr:template "https://khalilankri.../data/title/{title_id}";
        rr:class ont:Title
    ];
    rr:predicateObjectMap [
        rr:predicate ont:isClassifiedAs;
        rr:objectMap [
            rr:parentTriplesMap <#MapContentType>;  # Reference to TriplesMap
            rr:joinCondition [ 
                rr:child "content_type_id";   # FK in title table
                rr:parent "content_type_id"   # PK in content_type table
            ]
        ]
    ] .
Benefits:

Referential integrity: SQL JOIN ensures referenced entity exists
Proper typing: ContentType TriplesMap asserts rdf:type
Single source of truth: URI template defined once
Validation: If join fails (no matching row), no triple created

Result in RDF:
turtle<...contentType/2> a ont:ContentType ;
    ont:content_type_name "movie" .

<...title/tt001> a ont:Title ;
    ont:isClassifiedAs <...contentType/2> .

# Both entities properly typed!
# SPARQL query works correctly

Join Condition Explained
turtlerr:joinCondition [ 
    rr:child "content_type_id";    # Column in current table (title)
    rr:parent "content_type_id"    # Column in parent table (content_type)
]
SQL equivalent:
sqlSELECT t.title_id, ct.content_type_id
FROM title t
JOIN content_type ct ON t.content_type_id = ct.content_type_id
Processing:

R2RML processor reads title table
For each row, looks up matching content_type row
If match found: create triple linking the two IRIs
If no match: skip (referential integrity preserved)


1.4 NULL Handling
Problem: What happens when a foreign key is NULL?
Example:
sqltitle_aka table:
title_id | ord | aka_title        | region_id
tt001    | 1   | "El Padrino"     | 5
tt001    | 2   | "Le Parrain"     | 3
tt002    | 1   | "Batman Begins"  | NULL      ← No region
With direct template:
turtlerr:objectMap [ 
    rr:template "https://khalilankri.../data/region/{region_id}" 
]
⚠️ Problem: region_id=NULL produces malformed IRI: .../region/
With parent triples map:
turtlerr:objectMap [
    rr:parentTriplesMap <#MapRegion>;
    rr:joinCondition [ 
        rr:child "region_id"; 
        rr:parent "region_id" 
    ]
]
✓ Solution: NULL values cause join to fail gracefully → no triple created
Result:
turtle# Row with region_id=5
<...alternativeTitle/tt001_1> ont:isFrom <...region/5> .

# Row with region_id=3
<...alternativeTitle/tt001_2> ont:isFrom <...region/3> .

# Row with region_id=NULL
# NO ont:isFrom triple created (correct behavior!)

2. OUR MAPPING ARCHITECTURE
2.1 Namespace Strategy
Two-tier design:
Tier 1: Ontology Namespace (Schema)
https://khalilankri.github.io/audiovisual-ontology/1.0#
                                                     ↑
                                                   hash (#)
Used for:

Classes: ont:Title, ont:Talent, ont:Genre
Properties: ont:primaryTitle, ont:hasGenre, ont:participatesIn

Why hash (#)?

Fragment identifiers (client-side)
All ontology terms in single file
Standard for vocabularies/ontologies


Tier 2: Data Namespace (Instances)
https://khalilankri.github.io/audiovisual-ontology/data/
                                                     ↑
                                                  slash (/)
Used for:

Instances: data:title/tt001, data:genre/1, data:talent/nm0000123

Why slash (/)?

Each resource can have its own URL
RESTful access pattern
Enables content negotiation
Standard for instance data


Why separate namespaces?

Deployment flexibility:

Ontology served from /1.0/ontology.ttl
Data served from /data/titles.ttl, /data/talents.ttl, etc.


Content negotiation:

Request ont:Title → get class definition
Request data:title/tt001 → get instance data


Versioning:

Ontology: /1.0#, /2.0# (versions)
Data: /data/ (stays constant)


Best practices:

W3C Linked Data guidelines
Industry standard pattern




2.2 IRI Template Design
Principles:

One path segment per class:

data:title/tt001      ← /title/ segment
data:genre/1          ← /genre/ segment
data:talent/nm0000123 ← /talent/ segment
Prevents collisions between different entity types

Use natural IDs when available:

data:title/tt0068646    ← IMDb tconst (external identifier)
data:talent/nm0000199   ← IMDb nconst (external identifier)
Benefits:

Traceable to source data
Meaningful to humans
Stable across regenerations


Composite keys for junction tables:

data:participation/tt001_nm002_1
                   ↑      ↑      ↑
                   |      |      order number
                   |      talent_id
                   title_id
Ensures uniqueness when same entities appear multiple times

CamelCase for multi-word classes:

data:alternativeTitle/tt001_1    ← not alternative_title
data:contentType/2               ← not content_type
Matches ontology naming conventions

2.3 Dictionary Classes Pattern
What are dictionary classes?
Reference data, controlled vocabularies, lookup tables
Examples in our project:

Genre (Action, Drama, Comedy)
Category (actor, director, writer)
Role (character names)
ContentType (movie, TV series, short)
TitleType (original, alternative, DVD)
Region (country/region codes)

Common pattern:
turtle<#MapGenre> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "genre" ];
    rr:subjectMap [
        rr:template "https://khalilankri.../data/genre/{genre_id}";
        rr:class ont:Genre
    ];
    rr:predicateObjectMap [
        rr:predicate ont:genreName;
        rr:objectMap [ rr:column "genre_name" ]
    ];
    rr:predicateObjectMap [
        rr:predicate rdfs:label;
        rr:objectMap [ rr:column "genre_name" ]
    ] .
Why this pattern works:

Simple one-to-one mapping (1 row → 1 resource)
Single data property (name)
rdfs:label for human readability
Foundation for object properties in other entities

All 6 dictionary classes follow identical structure - code reuse, easy maintenance

Why complex?

Multiple data properties
Multiple object properties
Links to 3 different entity types
Central hub of the ontology

Breakdown:

    rr:logicalTable [ rr:tableName "title" ];

2.5 Participation Bridge Pattern
Challenge: Map many-to-many relationships
Relational model:
title (1) ←→ (N) title_principal (N) ←→ (1) talent
Semantic model: Reification (make the relationship a first-class entity)
turtleTitle --contains--> Participation <--participatesIn-- Talent

Why reification?

Rich metadata: Participation has its own properties:

ont:as → Category (actor, director, etc.)
ont:job → Job description
ont:plays → Role (character name)


Uniqueness: Same talent can appear multiple times in same title

Leonardo DiCaprio in "Inception" plays "Cobb" (lead actor)
Leonardo DiCaprio in "Inception" credited as "Executive Producer"
Two separate Participation entities needed!


Query flexibility: Can query:

All participations for a title
All participations by a talent
All participations in a specific category
Participation details (category, job, role)




Mapping:
turtle<#MapParticipation> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "title_principal" ];
    
    # Composite IRI ensures uniqueness
    rr:subjectMap [
        rr:template "https://khalilankri.../data/participation/{title_id}_{talent_id}_{ord}";
        rr:class ont:Participation
    ];
Why {title_id}_{talent_id}_{ord}?

title_id alone: Not unique (many talents per title)
talent_id alone: Not unique (talent in many titles)
title_id + talent_id: Not unique (talent can have multiple roles in same title!)
title_id + talent_id + ord: ✓ Unique!

Example:
title_id=tt001, talent_id=nm001, ord=1
→ data:participation/tt001_nm001_1

title_id=tt001, talent_id=nm001, ord=2
→ data:participation/tt001_nm001_2
Same talent, same title, different participation!

turtle    # Link to Category (actor, director, etc.)
    rr:predicateObjectMap [
        rr:predicate ont:as;
        rr:objectMap [
            rr:parentTriplesMap <#MapCategory>;
            rr:joinCondition [ 
                rr:child "category_id"; 
                rr:parent "category_id" 
            ]
        ]
    ];
Output: <data:participation/tt001_nm001_1> ont:as <data:category/1> .

turtle    # Job description
    rr:predicateObjectMap [
        rr:predicate ont:job;
        rr:objectMap [ rr:column "job" ]
    ] .
Output: <data:participation/tt001_nm001_1> ont:job "Lead Actor" .

Complete participation example:
Input (database row):
title_id: tt001
talent_id: nm001
ord: 1
category_id: 1
job: "Lead Actor"
Output (RDF):
turtle<data:participation/tt001_nm001_1> a ont:Participation ;
    ont:as <data:category/1> ;
    ont:job "Lead Actor" .

# Plus bidirectional links from Title and Talent:
<data:title/tt001> ont:contains <data:participation/tt001_nm001_1> .
<data:talent/nm001> ont:participatesIn <data:participation/tt001_nm001_1> .
Result: Rich, queryable network of relationships

2.6 The Role Problem & SQL JOIN Solution
Problem:
talent_role table structure:
talent_id | role_id
nm001     | 5
nm001     | 12
Missing: title_id and ord columns!
Why we need them:
Participation IRI requires: {title_id}_{talent_id}_{ord}
But talent_role only has: {talent_id}
Can't build Participation IRI! ❌

Solution: Custom SQL JOIN
turtle<#MapParticipationPlaysRole> a rr:TriplesMap;
    rr:logicalTable [ 
        rr:sqlQuery """
            SELECT tp.title_id, tp.talent_id, tp.ord, tr.role_id
            FROM title_principal tp
            JOIN talent_role tr ON tp.talent_id = tr.talent_id
        """
    ];
What this does:

Start with title_principal (has title_id, talent_id, ord)
JOIN talent_role (adds role_id)
Result: All 4 columns needed!

Query result:
title_id | talent_id | ord | role_id
tt001    | nm001     | 1   | 5
tt001    | nm001     | 1   | 12
tt002    | nm001     | 2   | 5

turtle    # Use SAME IRI as MapParticipation
    rr:subjectMap [
        rr:template "https://khalilankri.../data/participation/{title_id}_{talent_id}_{ord}"
        # NO rr:class! Participation already typed by MapParticipation
    ];
Key insight: Two TriplesMaps can target the same subject IRI!

MapParticipation creates the individual
MapParticipationPlaysRole adds properties to it


turtle    # Add ont:plays property
    rr:predicateObjectMap [
        rr:predicate ont:plays;
        rr:objectMap [
            rr:parentTriplesMap <#MapRole>;
            rr:joinCondition [ rr:child "role_id"; rr:parent "role_id" ]
        ]
    ] .
Output:
turtle# From MapParticipation:
<data:participation/tt001_nm001_1> a ont:Participation ;
    ont:as <data:category/1> ;
    ont:job "Lead Actor" .

# From MapParticipationPlaysRole:
<data:participation/tt001_nm001_1> ont:plays <data:role/5> .
<data:participation/tt001_nm001_1> ont:plays <data:role/12> .
Same subject, different properties!
One participation can have multiple roles - shows versatility (actor who also produces, directs, etc.)

2.7 Junction Table Pattern
What's a junction table?
Many-to-many relationship in relational databases
Examples:

title_genre: Links titles to genres (one title = many genres)
title_aka_title_type: Links alternative titles to types

Key difference from Participation:
Junction tables don't create new entities - they just add properties to existing ones

Example: title_genre
sqltitle_genre table:
title_id | genre_id
tt001    | 1        -- The Godfather → Action
tt001    | 3        -- The Godfather → Drama
tt002    | 5        -- Inception → Sci-Fi
Mapping:
turtle<#MapTitleGenre> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "title_genre" ];
    
    # Reconstruct existing Title IRI
    rr:subjectMap [
        rr:template "https://khalilankri.../data/title/{title_id}"
        # NO rr:class! Title already typed by MapTitle
    ];
    
    # Add ont:hasGenre property
    rr:predicateObjectMap [
        rr:predicate ont:hasGenre;
        rr:objectMap [
            rr:parentTriplesMap <#MapGenre>;
            rr:joinCondition [ rr:child "genre_id"; rr:parent "genre_id" ]
        ]
    ] .
Key points:

No rr:class - Title already has rdf:type ont:Title from MapTitle
Reconstructs IRI - Uses template matching MapTitle's pattern
Only adds properties - Just enriches existing resource

Output:
turtle# From MapTitle:
<data:title/tt001> a ont:Title ;
    ont:primaryTitle "The Godfather" .

# From MapTitleGenre:
<data:title/tt001> ont:hasGenre <data:genre/1> .
<data:title/tt001> ont:hasGenre <data:genre/3> .
Multiple rows → Multiple property values (multi-valued property)

2.8 Alternative Titles
Special case: New entity created from junction table
turtle<#MapAlternativeTitle> a rr:TriplesMap;
    rr:logicalTable [ rr:tableName "title_aka" ];
    
    # Composite IRI (title can have many akas)
    rr:subjectMap [
        rr:template "https://khalilankri.../data/alternativeTitle/{title_id}_{ord}";
        rr:class ont:AlternativeTitle
    ];
    
    # Alternative name
    rr:predicateObjectMap [
        rr:predicate ont:alternativeTitleName;
        rr:objectMap [ rr:column "aka_title" ]
    ];
    
    # Link to region (NULL-safe!)
    rr:predicateObjectMap [
        rr:predicate ont:isFrom;
        rr:objectMap [
            rr:parentTriplesMap <#MapRegion>;
            rr:joinCondition [ rr:child "region_id"; rr:parent "region_id" ]
        ]
    ];
    
    # Human-readable label
    rr:predicateObjectMap [
        rr:predicate rdfs:label;
        rr:objectMap [ rr:column "aka_title" ]
    ] .
Why composite IRI {title_id}_{ord}?

Same title can have multiple alternative titles
ord provides ordering/uniqueness

NULL handling:

If region_id is NULL, join fails gracefully
No ont:isFrom triple created (correct behavior)
Alternative title still created, just without region link

Back-link from Title:
Handled by MapTitle's ont:isAka predicate-object map
Result: Bidirectional relationship
turtle<data:title/tt001> ont:isAka <data:alternativeTitle/tt001_1> .
<data:alternativeTitle/tt001_1> ont:alternativeTitleName "El Padrino" ;
    ont:isFrom <data:region/5> .

2.9 rdfs:label Enhancement
Why rdfs:label?
Without labels:
sparqlSELECT ?title WHERE { ?title a ont:Title } LIMIT 5
Result:
https://khalilankri.github.io/audiovisual-ontology/data/title/tt0000001
https://khalilankri.github.io/audiovisual-ontology/data/title/tt0000002
https://khalilankri.github.io/audiovisual-ontology/data/title/tt0000005
Hard to interpret! Need to dereference each URI or do additional queries.
With labels:
sparqlSELECT ?label WHERE { 
    ?title a ont:Title ; 
           rdfs:label ?label 
} LIMIT 5
Result:
"1Zindagi"
"Im Schatten des Berges"
"Foreign Intrigue"
Immediately readable!

Implementation:
Added to every entity type:
turtlerr:predicateObjectMap [
    rr:predicate rdfs:label;
    rr:objectMap [ rr:column "primary_title" ]  # or talent_name, genre_name, etc.
]
Benefits:

SPARQL results: Readable without URI dereferencing
RDF browsers: Protégé, GraphDB show labels in UI
RDFS reasoning: Labels propagate in inference
Standard practice: W3C recommendation
Multilingual support: Can have multiple labels in different languages

Cost: One additional triple per entity (~10,000 extra triples)
Worth it? Absolutely! Massive usability improvement for minimal cost.

5. POTENTIAL QUESTIONS & ANSWERS
Q1: "Why use R2RML instead of writing custom code?"
Answer:

Standard: W3C Recommendation - interoperable, well-documented
Declarative: Easier to understand and maintain than imperative code
Tool support: Multiple processors available (our choice, Ontop, etc.)
Repeatability: Same mappings produce same output every time
Documentation: Mappings ARE the documentation (self-describing)
Efficiency: Processors optimized for large datasets

Custom code problems:

Requires programming expertise
Hard to maintain
No standard interchange format
Reinventing the wheel


Q2: "Why not use direct templates everywhere instead of parent triples maps?"
Answer:
Already covered in detail above, but key points:

Referential integrity: Parent triples maps enforce it like SQL foreign keys
No homeless individuals: Guarantees rdf:type assertions
Single source of truth: URI pattern defined once
NULL handling: Graceful failure instead of malformed IRIs
Validation: Can't create broken links

Trade-off: Slightly more complex syntax, but worth it for data quality

Q3: "How do you handle updates to the database?"
Answer:
Currently: Regenerate everything
bash# Update database
mysql -u root -p audiovisual_db < updates.sql

# Regenerate RDF
java -jar r2rml.jar config.properties

# Reload into Fuseki
# (Delete old data, upload new out.ttl)
For production: Could use:

Incremental mappings: Only map changed rows
Change tracking: Database triggers to track modifications
SPARQL UPDATE: Modify triples directly in triplestore
Versioning: Keep historical versions of data

Our choice: Full regeneration is fine for academic project (dataset is small, updates are infrequent)

Q4: "Why composite IRIs for Participation and AlternativeTitle?"
Answer:
Participation:

Same talent can appear multiple times in same title
Example: Actor who also produces → 2 participations needed
{title_id}_{talent_id} not enough → add {ord} for ordering

AlternativeTitle:

Same title can have many regional/language variations
{title_id} alone creates collision
{ord} provides uniqueness

Alternative approaches:

UUID: Random IDs → not traceable, not semantic
Hash of properties: Complex, fragile
Auto-increment: Requires database-generated IDs

Our choice: Composite keys are transparent, semantic, and stable

Answer:
Minor changes (add column):

Major changes (rename table, change foreign keys):

Update TriplesMap definitions

Regenerate
Validate with SHACL

Breaking changes (remove required column):

May need ontology changes too

Coordinate with data consumers

Best practice:

Document changes

3.2 Query Endpoint
Full implementation:
python@app.route('/query', methods=['POST'])
def query():
    try:
        # Get SPARQL query from request
        data = request.get_json()
        sparql = data.get('sparql', '')
        
        if not sparql:
            return jsonify({'error': 'No SPARQL query provided'}), 400
        
        # Forward to Fuseki
        response = requests.post(
            FUSEKI_ENDPOINT,
            data={'query': sparql},
            headers={'Accept': 'application/sparql-results+json'},
            timeout=30  # 30 second timeout
        )
        
        # Check Fuseki response
        if response.status_code != 200:
            return jsonify({
                'error': f'Fuseki error: {response.status_code}',
                'details': response.text
            }), response.status_code
        
        # Parse and format results
        fuseki_results = response.json()
        
        # Extract variable names
        vars = fuseki_results.get('head', {}).get('vars', [])
        
        # Extract bindings (rows)
        bindings = fuseki_results.get('results', {}).get('bindings', [])
        
        # Format for frontend
        rows = []
        for binding in bindings:
            row = {}
            for var in vars:
                if var in binding:
                    row[var] = binding[var].get('value', '')
                else:
                    row[var] = ''
            rows.append(row)
        
        return jsonify({
            'vars': vars,
            'rows': rows,
            'count': len(rows)
        })
    
    except requests.exceptions.Timeout:
        return jsonify({'error': 'Query timeout (>30s)'}), 504
    
    except requests.exceptions.ConnectionError:
        return jsonify({'error': 'Cannot connect to Fuseki. Is it running?'}), 503
    
    except Exception as e:
        return jsonify({'error': f'Server error: {str(e)}'}), 500
Key features:

Validation: Checks for empty query
Timeout: 30-second limit prevents hanging
Error handling: Different errors for different failure modes
Formatting: Converts Fuseki JSON to simpler format for frontend
HTTP status codes: Proper REST semantics

def execute_count_query(sparql):

function buildResultTable(vars, rows) {

    const thead = table.createTHead();
    const headerRow = thead.insertRow();
    vars.forEach(v => {

        th.textContent = v;
        headerRow.appendChild(th);

            td.textContent = row[v] || '';

Q5: "How would you add caching to improve performance?"
Answer:
Problem: Same query executed repeatedly → wasted computation
Solution 1: Flask-side caching
pythonfrom flask_caching import Cache

cache = Cache(app, config={'CACHE_TYPE': 'simple'})

@app.route('/query', methods=['POST'])
@cache.cached(timeout=300, query_string=True)  # 5min cache
def query():
    # ... existing code ...
Solution 2: Result caching
pythonimport hashlib

query_cache = {}

def execute_query(sparql):
    # Hash query for cache key
    cache_key = hashlib.md5(sparql.encode()).hexdigest()
    
    if cache_key in query_cache:
        return query_cache[cache_key]  # Cache hit
    
    # Execute query
    result = requests.post(...)
    
    # Store in cache
    query_cache[cache_key] = result
    
    return result
Trade-offs:

✅ Faster response for repeated queries
✅ Reduced load on Fuseki
❌ Stale data if RDF updates
❌ Memory usage for cache

For our demo: Not needed (queries fast enough, data static)
